## 2024-05-09 - Path Traversal in Image Filenames
**Vulnerability:** The API returned attribute `FileName` was used without any sanitization in `get_image_filename()`. An attacker controlling the SmugMug API or the payload could provide `FileName: "../../../etc/passwd"`, causing the engine to write the downloaded file outside of the intended album download directory, which leads to path traversal.
**Learning:** External API data should never be trusted when constructing file paths on the local file system. Even if it is a 3rd party API, we must sanitize paths (such as `FileName`) locally before downloading them.
**Prevention:** Use `os.path.basename(filename.replace("\\", "/"))` to strictly extract just the intended filename component of the provided payload, blocking both Unix and Windows directory traversal characters.

## 2024-05-13 - Path Traversal Bypass via Empty Basename
**Vulnerability:** The fix for path traversal in `get_image_filename()` used `os.path.basename` to extract a safe filename. However, if an attacker provided "." or ".." as the `FileName` parameter, `os.path.basename` would simply return "." or "..", allowing them to traverse up the directory tree or pollute the directory with non-image names.
**Learning:** `os.path.basename` is not always sufficient for stripping directory traversal strings because "." and ".." are considered valid file basenames in Unix paths. Always sanitize and reject these specific paths after extracting the basename.
**Prevention:** In addition to using `os.path.basename`, explicitly check `if filename in (".", "..")` and reject those values to ensure only valid filenames are created.

## 2026-05-14 - TOCTOU Vulnerability in OAuth Token Storage
**Vulnerability:** In `src/auth.py`, `save_tokens` saved OAuth tokens containing secrets to a file using the standard `open(TOKEN_FILE, "w")` followed by `os.chmod(TOKEN_FILE, 0o600)`. This created a Time-Of-Check to Time-Of-Use (TOCTOU) race condition where the token file was briefly written with default system permissions (like `0o644`) before being restricted.
**Learning:** Saving sensitive files by first creating them with wide permissions and modifying them afterwards is insecure, as an attacker or unauthorized user on the system might read the token during the brief window before `chmod` is called.
**Prevention:** Use `os.open` with the flags `os.O_CREAT | os.O_WRONLY | os.O_TRUNC` and explicit `0o600` mode to ensure the file is created with secure permissions right from the start.

## 2026-05-15 - Path Traversal bypass via Fallback Image Key
**Vulnerability:** A previous fix addressed a Path Traversal issue in `get_image_filename()` by sanitizing `FileName` via `os.path.basename`. However, the code would fallback to using `extract_image_key(image)` if `FileName` was missing or became empty after sanitization. This fallback `ImageKey` was used directly in string interpolation for the filename (`f"{key}.{ext}"`) without any sanitization, allowing path traversal attacks via the `ImageKey` field.
**Learning:** Fallback data sources (like `ImageKey` used when `FileName` is missing) can also contain malicious payloads. Any external data used to construct file paths must be sanitized, regardless of whether it's the primary source or a fallback.
**Prevention:** Apply the exact same path traversal sanitization (using `os.path.basename` and explicitly handling "." / "..") to the fallback `ImageKey` before utilizing it to construct a local file path. Provide a safe default (like "untitled_image") if the key is effectively empty.

## 2024-05-16 - SSRF and Token Leak in API Client
**Vulnerability:** The `download_file` method in `src/api_client.py` accepted any URL passed to it (e.g. from an API response) and made an HTTP GET request using an authenticated `OAuth1Session`. This session automatically attaches the OAuth `Authorization` header. If a malicious URL (like `http://attacker.com/`) was provided, the application would perform a Server-Side Request Forgery (SSRF) and leak the sensitive OAuth credentials to the attacker's server.
**Learning:** Authenticated sessions (like `requests_oauthlib.OAuth1Session` or standard `requests.Session` with global auth) will blindly send sensitive authentication headers to any domain they are told to request. You cannot trust URLs originating from external sources, even API responses.
**Prevention:** Always parse and strictly validate URLs before making requests with an authenticated session. Enforce the `https` scheme and verify that the hostname strictly matches the expected domain or its subdomains (e.g. `smugmug.com`).

## 2026-05-18 - SSRF and OAuth Token Leak via Open Redirects and Pagination
**Vulnerability:** The API client automatically followed HTTP redirects using `requests` with an authenticated `OAuth1Session`. If the SmugMug API returned an open redirect (e.g. to `https://attacker.com`), `requests_oauthlib` would leak the sensitive OAuth credentials to the attacker's server by attaching the `Authorization` header to the redirected request. Additionally, the pagination logic blindly constructed URLs from the `NextPage` attribute, which could lead to SSRF if the API returned an endpoint starting with `@attacker.com`.
**Learning:** Initial URL validation is not enough to prevent SSRF and token leaks; you must also securely handle or disable HTTP redirects, as HTTP clients typically re-attach credentials to cross-host redirects. Furthermore, relative path API inputs must be strictly validated to prevent hostname injection.
**Prevention:** Disable automatic redirects (`allow_redirects=False`) and manually resolve and validate the `Location` header against a strict domain whitelist before following them. Validate that relative API endpoints strictly start with `/` to prevent hostname injection.
## 2024-05-18 - SSRF and OAuth Token Leak via API Client Redirects
**Vulnerability:** The API client automatically followed HTTP redirects using requests with an authenticated OAuth1Session in the `_request` method. If the SmugMug API returned an open redirect (e.g., to https://attacker.com), requests_oauthlib would leak the sensitive OAuth credentials to the attacker's server by attaching the Authorization header to the redirected request.

## 2024-05-20 - SSRF Bypass via URL Parsing Discrepancy
**Vulnerability:** The SSRF protection in the API client validated URLs by verifying that `urllib.parse.urlparse(url).hostname` ended with `.smugmug.com`. However, there is a parsing discrepancy between Python's standard `urlparse` and the `requests` library (which actually executes the HTTP requests). For a URL like `https://attacker.com\@smugmug.com/`, `urlparse` extracts `smugmug.com` as the hostname, passing the validation. But `requests` prepares the URL as `https://attacker.com/%5C@smugmug.com/`, extracting `attacker.com` as the true target hostname. This allows an attacker to bypass the domain restriction, causing SSRF and leaking the OAuth token to `attacker.com`.
**Learning:** You cannot safely use `urlparse` to validate the hostname of a URL that will be passed directly to another HTTP client library if the two parse edge cases differently.
**Prevention:** Always normalize the URL using the HTTP client's own mechanism *before* validation. By running `p = PreparedRequest(); p.prepare_url(url, None)` and validating `p.url`, you ensure that the URL being validated matches exactly what the HTTP client will request.

## 2024-05-25 - TOCTOU and Symlink Vulnerability in File Downloads
**Vulnerability:** The `download_file` method in `src/api_client.py` created temporary files using predictable paths (`dest_path + ".tmp"`) and the standard `open(..., "wb")` function. It also finalized the download with `os.rename()`. This introduced a Time-Of-Check to Time-Of-Use (TOCTOU) race condition and potential for symlink attacks, as an attacker with local access could predict the temp file name and manipulate it before or during the write process.
**Learning:** Temporary files should never be created using predictable names or standard `open()` because of inherent race conditions and symlink vulnerabilities. Standard file rename functions like `os.rename()` may not guarantee an atomic replacement.
**Prevention:** Always use `tempfile.mkstemp(dir=..., suffix=...)` paired with `os.fdopen(fd, "wb")` to ensure unique, unpredictable, and atomically created temporary files with restrictive permissions. Finish file transfers by explicitly using `os.replace()` for an atomic overwrite.

## 2024-05-28 - Insecure MD5 Hash Usage
**Vulnerability:** The application used `hashlib.md5()` without explicitly marking it as not used for security (`usedforsecurity=False`), causing potential crashes in FIPS-compliant environments where MD5 is restricted.
**Learning:** When using deprecated hashing algorithms like MD5 strictly for non-security purposes (like file integrity checks), they must be explicitly flagged to prevent runtime errors on hardened systems.
**Prevention:** Always use `hashlib.md5(usedforsecurity=False)` when verifying file checksums to ensure compatibility with FIPS mode while maintaining security compliance.

## 2025-02-28 - [Terminal Output Injection via Rich library]
**Vulnerability:** The codebase used the `rich` library (`console.print`, `Table.add_row`, `Progress.add_task`) to display user-controlled inputs (API responses, filenames, album names, exception messages) directly without escaping. Because `rich` supports markup (like `[red]`), an attacker could craft malicious input to cause terminal injection, corrupt display, or hide text.
**Learning:** High-level terminal output libraries often support markup by default. Unsanitized data passed directly to these libraries is susceptible to terminal output injection attacks.
**Prevention:** Always use `escape` from `rich.markup` to sanitize all external or user-controlled inputs before passing them into display functions that evaluate markup.

## 2025-02-28 - Terminal Output Injection via Rich library (Part 2)
**Vulnerability:** Several files in the codebase (`src/api_client.py`, `src/auth.py`, `src/cli.py`) still passed unsanitized external variables (such as URLs, hostnames, exception messages, and output directories) to `rich.console.Console.print()`. This allows malicious input containing `rich` markup tags (like `[red]`) to corrupt terminal output or execute formatting injections.
**Learning:** Fixing injection vulnerabilities must be done comprehensively across the entire codebase. Every instance of an external or user-controlled variable passed to a markup-evaluating display function must be properly escaped.
**Prevention:** Apply `rich.markup.escape()` consistently to all untrusted variables before they are interpolated into strings used by `rich` display functions. Ensure that comprehensive tests verify its presence across all relevant modules.
